# Story 1.2: 选择并初始化本地工作目录

Status: done

## Story

As a 本地文档处理用户,
I want 在桌面应用中选择或切换一个本地工作目录,
so that slicer 可以在我指定的位置创建标准工作区结构，并在下次启动时恢复到可信的本地工作状态。

## Acceptance Criteria

1. Given 用户首次打开 slicer 且尚未配置工作区, when 用户在工作台或设置页选择一个本地目录, then 应用应初始化该目录为 slicer 工作区, and 工作台、设置页与侧栏状态应显示已选择的工作区路径。
2. Given 用户选择一个空目录或尚未完整初始化的目录, when 初始化工作区, then 应创建或确认基础结构: `originals/`, `pages/`, `analysis/`, `metadata/`, `indexes/`, `indexes/bm25/`, `jobs/`, `logs/`, `tmp/`, `app.db`, and 不得创建与故事范围无关的导入、分析、索引或任务数据。
3. Given 用户选择一个已经有效的 slicer 工作区或重复选择同一目录, when 初始化逻辑再次运行, then 操作必须幂等, and 不得删除、覆盖或清空既有文件、数据库、索引目录或用户资料。
4. Given 工作区初始化成功, when 应用关闭后重新启动, then 应用应恢复最近一次使用的工作区路径, and 如果该路径仍可访问则状态为可用。
5. Given 最近一次工作区路径缺失、被移动、不可访问或权限不足, when 应用启动或查询工作区状态, then 应返回可恢复的错误状态, and UI 应以中文显示原因摘要和重新选择入口，不得崩溃或卡在加载态。
6. Given 用户选择文件路径、不可写目录、受限目录或无法规范化的路径, when 初始化工作区失败, then 应返回结构化错误和中文用户提示, and 不得留下随机临时文件或无法识别的半初始化状态。
7. Given 后续功能需要使用 workspace 内部路径, when 代码请求 SQLite、originals、pages、analysis、metadata、indexes、jobs、logs 或 tmp 位置, then 这些路径必须由 Rust service/artifact/AppState 基础能力集中提供, and 前端不得拼接内部路径或假设 artifact 布局。
8. Given 前端需要选择目录, when 用户点击“选择工作区”或“更改工作区”, then 前端只负责打开目录选择器、传递用户选中的根目录并渲染 DTO 状态, and 目录验证、结构创建、持久化和恢复必须在 Rust 层完成。
9. Given Tauri command 或未来 HTTP API 需要返回工作区状态, when DTO 被序列化, then 对外字段使用 `snake_case`, and 状态值应稳定可测，例如 `not_selected`, `loading`, `ready`, `missing`, `invalid`, `error`。
10. Given 本故事建立工作区基础能力, when 开发者实现路径、状态或测试夹具, then 不得把 `page_id` 与 `image_hash` 混用或提前定义为同一个值；本故事不实现页面身份算法。

### Workspace Layout Contract

Rust layout capability should expose at least:

```text
workspace/
  originals/
  pages/
  analysis/
  metadata/
    pages.jsonl      # optional empty placeholder in this story; real export later
  indexes/
    bm25/
  jobs/
  logs/
  tmp/
  app.db
```

Implementation guidance:

- `WorkspaceLayout::from_root(root: PathBuf)` or equivalent should be the only place constructing internal paths.
- `WorkspaceService` should expose capability methods for Rust callers, such as `current_layout()`, `app_db_path()`, `originals_dir()`, `pages_dir()`, `analysis_dir()`, `metadata_dir()`, `indexes_dir()`, `bm25_index_dir()`, `jobs_dir()`, `logs_dir()`, `tmp_dir()`.
- The frontend may display the selected root path, but should not receive or construct page/index/database paths for future business operations.

### DTO Guidance

Suggested command DTO shape:

```ts
export type WorkspaceStatusValue =
  | "not_selected"
  | "loading"
  | "ready"
  | "missing"
  | "invalid"
  | "error";

export interface WorkspaceErrorDto {
  code: string;
  message: string;
  retryable: boolean;
  stage: "restore" | "select" | "initialize" | "validate";
  details?: string;
}

export interface WorkspaceStatusDto {
  status: WorkspaceStatusValue;
  workspace_path?: string;
  error?: WorkspaceErrorDto;
}
```

Notes:

- Keep serialized field names `snake_case`. TypeScript can use the same DTO names even if local UI helpers map values to Chinese copy.
- Do not return raw Rust filesystem errors directly to the UI. Map them to stable codes such as `workspace_not_selected`, `workspace_missing`, `workspace_not_directory`, `workspace_not_writable`, `workspace_initialize_failed`.
- Avoid leaking sensitive or excessive system details in user-visible `message`; diagnostic `details` should still be safe and concise.

  - Rename/delete the workspace directory, restart, confirm recoverable error and reselect entry.

### Anti-Patterns to Avoid

- Do not concatenate paths in React, `tauriClient`, Tauri command handlers, or string templates.
- Do not store the authoritative workspace path in frontend `localStorage`.
- Do not overwrite an existing `app.db`, `metadata/pages.jsonl`, `indexes/`, or any user file during idempotent initialization.
- Do not implement SQLite migrations, job tables, document/page status enums, import pipeline, BM25 provider, model provider, or localhost API in this story.
- Do not return raw OS errors or stack traces directly to the UI.
- Do not add fake directory selection via text input when the official Tauri dialog plugin is unavailable.
- Do not change the Story 1.1 navigation, logo, basic pages, or visual style except where needed to display live workspace status.
