## 1. Epic Review

### Epic 摘要

Epic 5 的目标是让 slicer 提供默认仅监听 `127.0.0.1` 的 localhost HTTP API，使本机脚本或工具可以查询健康状态、搜索结果、页面、文档，并通过 token 保护的写操作触发索引重建。

从 `_bmad-output/implementation-artifacts/sprint-status.yaml` 看，Epic 5 以及 6 个故事均已标记为 `done`:

1. Story 5.1: 嵌入式 Axum Localhost API 服务基础与设置控制
2. Story 5.2: 统一 HTTP DTO、成功响应与 AppError 错误映射
3. Story 5.3: `GET /health` 返回应用、工作区与索引状态
4. Story 5.4: `GET /search`、`GET /pages/{page_id}` 与 `GET /documents/{document_id}` 读取接口
5. Story 5.5: `POST /indexes/rebuild` token 保护与后台重建任务
6. Story 5.6: API 合同测试、Settings 可见性与外部访问收口

当前代码侧也能看到完整端点已经注册在 `src-tauri/src/api/server.rs`:

- `GET /health`
- `GET /search`
- `GET /pages/{page_id}`
- `GET /documents/{document_id}`
- `POST /indexes/rebuild`

### 做得好的地方

1. API 边界按架构落位

`src-tauri/src/api/server.rs` 只负责路由和 server，业务处理进入 `api/endpoints.rs` 后再调用 `SearchService`、`DocumentRepository` 等既有内部能力。整体方向符合架构文档中的 “HTTP API -> API routes -> Services -> Repositories/Artifacts/Providers” 流向，没有形成独立的 HTTP-only 业务分叉。

2. 安全默认值贯彻得较好

Epic 5 明确保持 localhost-only 默认绑定，并有 `rejects_non_loopback_bind_address` 覆盖 `0.0.0.0` 拒绝场景。重任务端点 `POST /indexes/rebuild` 通过 Bearer token 保护，缺失 token 和错误 token 都有合同测试。

3. 响应契约统一

Story 5.2 引入 `{ "data": ... }` 与 `{ "error": ... }` 结构，并把 `AppError` 映射到 HTTP 错误响应。后续端点复用这一层，降低了外部脚本解析成本。

4. 测试策略逐步成熟

Story 5.1 使用动态端口避免端口冲突，Story 5.6 增加了端点级合同测试。回顾期间补跑结果显示 Rust 单元测试与 API 集成测试均通过。

5. Settings UI 形成了可操作入口

设置页已经展示 API 启停、监听地址、端口、运行状态、token reset、端点摘要。对于本地自动化用户，这比只提供隐藏接口更符合可发现性要求。

### 遇到的挑战

1. 故事文件状态与 sprint-status 不一致

`sprint-status.yaml` 中 5.1、5.3、5.4 都是 `done`，但对应 story/spec 文件里仍有 `status: review` 或 `**Status:** review`。这不一定代表代码未完成，但会削弱后续自动化对状态的信任。

2. 合同测试更偏 “形状验证”，真实成功路径覆盖不足

API 合同测试对 `GET /search`、`GET /pages/{id}`、`GET /documents/{id}`、有效 token rebuild 等场景允许 `500` 作为可接受结果，只要求错误契约成立。这对早期端点合同是合理取舍，但还不足以证明在真实初始化工作区、有文档、有索引的情况下业务成功路径完全可靠。

3. token 生命周期仍偏 MVP 简化

当前 `ApiAppState` 中的 `api_token` 是内存态，应用启动时生成，`reset_api_token` 返回新 token。设置页显示截断 token，但没有持久化、一次性复制策略或重启后 token 迁移策略说明。对于本机 MVP 可以接受，但对稳定外部自动化脚本会产生配置更新成本。

4. API 文档仍主要嵌在 UI 中

Story 5.6 明确独立 API 使用文档 out of scope，端点摘要只在 Settings UI 中。对本地脚本用户来说，UI 摘要足够发现端点，但不够支撑长期自动化集成和复制示例。

5. 前端验证入口受本机环境影响

本次回顾无法通过 `npm run build` 复核前端构建。故事记录显示过往曾运行 `npx tsc --noEmit` / `npx vite build`，但回顾时当前环境没有完成这一层再验证。

### 关键学习

1. 先建生命周期，再建 DTO，再建业务端点，是正确的拆分顺序

Story 5.1 到 5.3 的分层推进让 API 从 “能启动” 到 “能统一响应” 再到 “能返回业务状态” 逐步稳定，避免了在一开始同时处理 server 生命周期、错误契约和业务查询。

2. `AppError` 统一模型继续产生复利

Epic 1.5 的错误模型在 Epic 5 中直接支撑 HTTP 错误响应、端口失败、认证失败和安全脱敏。这个基础设施没有被绕开，是本 Epic 质量稳定的关键。

3. 合同测试需要分层

当前合同测试证明了 endpoint shape、auth failure、error body shape。下一步需要补 “真实数据成功路径” 层，否则外部自动化用户最关心的 happy path 仍然只被间接覆盖。

4. 状态账本必须成为自动化可信源

故事文件和 sprint-status 的状态不一致说明收口步骤还需要自动化或 checklist 支撑。否则回顾、自动 story 选择和 release 判断都会受到噪声影响。

## 3. Next Epic Preparation

规划文档中没有 Epic 6；PRD 的下一阶段目标是 M5 MVP 打磨与验收。Epic 5 完成后，产品已经具备 M4 “检索与 API” 的主体能力，接下来应围绕 MVP 可验收性做收口。

PRD 中 M5/MVP 验收重点包括:

- Windows 环境启动、工作目录选择和设置持久化。
- PDF、PPTX、DOCX 样例文件端到端流程。
- 30 页 PPTX 与 300 页 PDF 验收。
- 中文路径、长文件名、路径含空格。
- 应用重启恢复验证。
- 搜索返回页面 JSON、图片地址、来源文档和页码。
- localhost HTTP API 可查询。
- 转换/分析失败可见且可重试。
- 索引可重建，失败时不破坏已有元数据。

### 对下一阶段的影响

Epic 5 不要求重写下一阶段方向，但它暴露出下一阶段需要更重视 “端到端验收矩阵” 而不是单 endpoint 实现。

重点准备项:

1. 准备真实验收工作区

包含 PDF、PPTX、DOCX 样例，覆盖中文路径、长文件名和含空格路径。这个工作区应能支撑 GUI 与 localhost API 的共同验收。

2. 补 API 成功路径合同测试

基于真实初始化工作区或测试 fixture，覆盖:

- `/health` 在 ready workspace 下包含 index 状态。
- `/search` 在已构建索引后返回真实结果。
- `/pages/{page_id}` 返回真实页面 JSON 和来源字段。
- `/documents/{document_id}` 返回真实文档元数据和页面列表。
- `/indexes/rebuild` 有效 token 触发真实任务，并能和 GUI 重建语义保持一致。

3. 收口 token 策略

明确 token 是否只在内存中有效，外部脚本如何获取/更新 token，重启后旧 token 失效是否符合产品预期。若保持 MVP 简化，需要在 UI 和文档中说清楚。

4. 补独立 API 使用说明

至少提供本地 markdown 或设置页可复制示例，覆盖 endpoint、认证方式、响应结构、常见错误码和脱敏原则。

5. 清理状态账本不一致

将 Epic 5 对应 story/spec 文件状态与 `sprint-status.yaml` 对齐，避免后续自动化误判。

6. 恢复前端验证命令

让本地环境可以稳定执行前端类型检查和构建命令，并把失败原因从 PATH/工具链问题中剥离出来。
