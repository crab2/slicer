# Story 3.4: 新页面批量分析与单文档重新分析

**Status:** done

## Story

As a 本地文档处理用户,  
I want 对新页面批量执行分析，并能对单个文档重新分析,  
So that 我可以高效处理导入后的页面，同时在 prompt 或模型配置变化后刷新某个文档的分析结果。

## Acceptance Criteria

1. **Given** 工作区中存在多个已渲染但尚未分析的页面  
   **When** 用户点击分析入口或系统启动分析任务  
   **Then** 应用应只选择新页面或被明确标记为需重跑的页面进入分析队列  
   **And** 已有有效分析结果的页面不得被无理由重复调用模型

2. **Given** 用户选择对单个文档重新分析  
   **When** 应用创建重新分析任务  
   **Then** 系统应将该文档下所有页面标记为待分析或创建新的分析版本  
   **And** 不应影响其他文档的有效分析结果

3. **Given** 批量分析任务正在运行  
   **When** 多个页面等待分析  
   **Then** Job Orchestrator 应报告总页数、已完成页数、失败页数、当前阶段和最近更新时间  
   **And** 并发度应遵守设置页中的分析并发数

4. **Given** 批量分析中部分页面成功、部分页面失败  
   **When** 任务结束或暂停  
   **Then** 成功页面应保留已校验的分析结果  
   **And** 失败页面应保留失败状态和可重试信息，不得导致整个文档所有页面回滚为失败

5. **Given** 应用关闭或崩溃时存在运行中的分析任务  
   **When** 用户重新打开同一工作区  
   **Then** 系统应通过 SQLite 识别未完成分析任务和页面状态  
   **And** 不确定状态应被标记为可恢复失败、可重试或重新排队，不能永久显示运行中

6. **Given** 批量分析任务完成  
   **When** 用户查看工作台任务列表或文档摘要  
   **Then** 应显示文档分析进度、成功页数、失败页数和最近分析时间  
   **And** 界面应为失败页面提供单独重试入口

- [x] **补齐批量分析查询与任务 DTO**（AC: 1, 3, 6）
  - [x] 在 `DocumentRepository` 或专门的查询 repository 中新增可复用查询：列出所有待分析页面、列出指定文档页面、按页面状态和当前 `analysis_results.status` 判断是否需要分析
  - [x] 新页面选择规则：默认只选择 `page_records.status` 为 `rendered` 或 `failed` 且没有当前 `succeeded` 分析结果的页面；`analysis_pending` 不得被重复入队
  - [x] 单文档重跑规则：只选择目标 `document_id` 下的页面；允许覆盖该文档页面的 current analysis result，但不得选择其他文档页面
  - [x] 新增 `AnalysisBatchResultDto` 或等价 DTO，至少返回 `job_id`、`total_pages`、`succeeded_pages`、`failed_pages`、`skipped_pages`、`status`、`updated_at`
  - [x] 如果要在 job event message 中承载计数，保持安全摘要格式，不写入 API key、Authorization、完整图片、完整模型响应或未脱敏诊断

- [x] **重构 `AnalysisService` 以复用单页核心逻辑**（AC: 1–4）
  - [x] 保留现有 `analyze_page` 对外行为，抽出内部 helper 处理“单页准备、provider 调用、schema 校验、成功/失败持久化”，让批量流程复用同一实现
  - [x] 新增 `analyze_new_pages(workspace)` 或等价 service 方法：创建一个 `page_analysis_batch` job，读取模型配置、隐私状态和 `analysis_concurrency`，只处理 eligible 新页面
  - [x] 新增 `reanalyze_document(workspace, document_id)` 或等价 service 方法：验证文档存在，创建 `document_reanalysis` job，仅处理该文档页面
  - [x] 当前 `analysis_results` 表是 `UNIQUE(page_id)` current-result 模型；本故事可复用 `save_success_result` 覆盖 current result，不要提前实现 Story 3.6 的版本历史或 JSONL 导出
  - [x] 并发执行不得让同一页面同时被两个分析任务处理；继续使用或增强 `analysis_pending` 作为 lease/互斥，较慢旧请求不得覆盖较新结果
  - [x] provider 调用期间不得持有 SQLite connection；批量执行每页持久化必须独立收敛，单页失败不能回滚其他页面成功结果

- [x] **实现批量进度、失败收敛与崩溃恢复**（AC: 3–5）
  - [x] Job 进度按页面总数计算，消息中包含当前阶段、已完成页数、失败页数、跳过页数和最近处理页面
  - [x] 部分失败时，batch job 可最终标记为 `succeeded` with failures 或 `failed`，但必须有清晰规则；无论 job 最终状态如何，成功页面保留 `analyzed` 和 `succeeded` result
  - [x] 失败页面必须保存 `page_records.status = failed`、`error_summary` 和 failed `analysis_results`（含 `error_id`），便于 Story 3.5 单页重试继续使用
  - [x] `recover_interrupted_jobs` 目前只恢复 job；本故事必须同步处理遗留 `analysis_pending` 页面，将其标记为可恢复失败或重新排队，避免页面永久“分析中”
  - [x] 配置缺失、隐私未确认等前置条件失败不得污染页面状态；只有已经获得页面 lease 并进入分析流程的页面才可被标记失败

- [x] **接入 Tauri commands 与前端工作台批量入口**（AC: 1, 2, 6）
  - [x] 在 `analysis_commands.rs` 新增 `analyze_new_pages` 与 `reanalyze_document(document_id)`；command 只做参数映射、`spawn_blocking` 和调用 service
  - [x] 在 `lib.rs` / `commands/mod.rs` 注册新 command；所有 DTO 字段保持 `snake_case`
  - [x] 更新 `src/lib/tauriClient.ts` 和 `src/types/app.ts`，为批量分析与单文档重分析增加 typed client 方法和 DTO 类型
  - [x] 更新 `WorkbenchPage.tsx` 的“模型分析”区域：模型配置和隐私门禁通过后，主按钮启动新页面批量分析，而不是只显示就绪提示
  - [x] 更新 `DocumentList.tsx`：对文档展示“重新分析此文档”入口；对失败页面保留单页分析/重试入口；已成功分析页面只有在文档重分析或明确单页重试时才重新调用模型
  - [x] 工作台刷新必须从后端查询 jobs/documents/pages 恢复状态，不得把 React 内存里的进度当作 source of truth

| 能力 | 当前实现 | Story 3.4 动作 |
|------|----------|----------------|
| 单页分析 | `AnalysisService::analyze_page_with_provider` 已完成配置/隐私校验、创建 `page_analysis` job、读取页面图片、调用 provider、校验 schema、保存结果 | 不要复制一套批量分析逻辑；抽出内部 helper，批量和单页共用 provider/validator/persistence |
| 分析结果账本 | `analysis_results` 当前是 `UNIQUE(page_id)`，`AnalysisRepository::save_success_result` / `save_failure_result` 覆盖 current result | 本故事继续使用 current-result 模型；版本历史与 JSONL 输出留给 Story 3.6 |
| 页面状态 | `page_records.status` 允许 `rendered`、`analysis_pending`、`analyzed`、`failed`；单页分析用 `analysis_pending` 防重复 | 批量任务必须以该状态作为 lease/互斥基础，避免同页重复 provider 调用 |
| 页面/文档查询 | `DocumentRepository` 已有 `find_page_by_id`、`find_document_by_id`、`find_image_asset_by_hash`、`list_pages_by_document`、`list_all_pages` | 追加“eligible pages”查询，不要让 service 手写散落 SQL |
| 设置 | `SettingsService::get_settings` 提供 `analysis_concurrency`；`get_model_configuration_status` 提供配置和隐私门禁 | 批量分析必须遵守 `analysis_concurrency`，并复用配置/隐私门禁 |
| Job | `JobOrchestrator` 支持 `create_job`、`update_progress`、`mark_failed`、`recover_interrupted_jobs`；进度 message 是当前最直接的摘要承载点 | 批量 job 类型建议为 `page_analysis_batch` 和 `document_reanalysis`；必要时扩展 DTO，但不要破坏现有 job 列表 |
| 前端 | 工作台已有单页分析入口、隐私弹窗、`analyzePage(pageId)` client 和文档/页面刷新 | 将模型分析主入口升级为批量分析；文档卡片增加单文档重分析入口 |

### 实现边界

- 本故事只做批量分析、单文档重新分析、批量进度和工作台入口，不实现 Story 3.5 的高级失败分类策略、批量失败项重试策略、配置修复阻断策略。
- 本故事不实现 Story 3.6 的分析版本历史、页面 JSON/JSONL artifact 导出、当前有效结果指针重构。
- 本故事不实现 Epic 4 的 BM25 索引构建、搜索 UI、索引重建，也不实现 Epic 5 localhost HTTP API。
- 不新增真实外部模型测试依赖；service 测试继续使用 mock provider。

### Previous Story Intelligence

- Story 3.3 code review 已修复关键失败收敛问题：页面读取失败、provider 失败、schema 校验失败会通过 job/page failure handling 收敛；批量实现不能绕开该路径。
- Story 3.3 已增加 custom HTTP provider 的超时、响应大小、HTTPS 和安全错误摘要保护；批量实现不得为了并发调用直接构造 `reqwest` 或绕开 `ModelProvider`。
- Story 3.3 已确认配置缺失或隐私未确认不应把页面标记为 failed；批量前置校验也必须保持这个行为。
- Story 3.2 validator 会严格校验 `page_id`、`image_hash`、`image_path` 和 source；批量重分析仍必须逐页构造准确 `ExpectedPageContext`。
- Story 3.1 的 keyring 与隐私确认是模型调用的安全边界；前端批量按钮不能绕过设置状态或直接接触明文 key。
