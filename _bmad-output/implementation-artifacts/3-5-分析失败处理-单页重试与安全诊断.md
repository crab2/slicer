# Story 3.5: 分析失败处理、单页重试与安全诊断

**Status:** done

## Story

As a 本地文档处理用户,  
I want 非法 JSON、超时、API 错误或网络失败的页面分析能被清楚记录并单独重试,  
So that 我可以修复配置或服务问题后继续处理失败页面，而不用重做全部文档。

## Acceptance Criteria

1. **Given** 模型调用发生超时、网络错误、HTTP 错误、provider 错误、非法 JSON 或 schema 校验失败  
   **When** `AnalysisService` 捕获失败  
   **Then** 页面分析状态应更新为失败或待重试状态  
   **And** 任务、页面和 `analysis_results` failed current result 应关联结构化 `AppError`，包含 `stage`、`retryable`、`correlation_id` 和中文安全摘要

2. **Given** 失败原因包含 API response、headers、endpoint、prompt 摘要或模型输出片段  
   **When** 系统写入日志、错误详情、job event、page `error_summary` 或诊断摘要  
   **Then** 应执行 redaction，移除 API key、Authorization header、token 和类似 secret 字段  
   **And** 原始模型响应只允许以安全截断摘要保存，不得完整写入普通日志、SQLite 错误详情、job event、前端状态或 JSONL

3. **Given** 某个页面分析失败且被标记为可重试  
   **When** 用户点击单页重试  
   **Then** 应用应只为该页面创建新的分析尝试  
   **And** 其他页面的有效分析结果不得被覆盖、删除或重新调用模型

4. **Given** 失败原因是配置缺失、API key 无效、隐私提示未确认或 endpoint 不可达  
   **When** 用户查看失败页面或任务  
   **Then** 界面应提示用户检查模型设置、隐私确认、网络或 provider 状态  
   **And** 在配置修复前可以阻止继续批量重试，避免重复产生同类失败

5. **Given** 重试后模型返回有效 `page_analysis_v1` JSON  
   **When** schema 校验通过并入库  
   **Then** 页面状态应从失败变为已分析  
   **And** 历史失败记录应保留用于诊断，但当前有效分析结果应明确可识别

6. **Given** 开发者编写测试  
   **When** 使用 mock/test provider 模拟超时、非法 JSON、HTTP/API 错误、schema 校验失败和成功重试  
   **Then** 每类失败都应映射到可测试的错误 code、stage 和 retryable 行为  
   **And** 测试不得需要真实外部模型服务、真实 API key 或真实网络

### 实现边界

- 本故事只处理分析失败分类、单页重试、安全诊断、配置修复提示和相关测试。
- 本故事不实现 Story 3.6 的页面 JSON/JSONL 生成、分析版本历史、artifact 原子写入。
- 本故事不实现 Story 3.7 的完整工作台分析体验收口、全部失败项重试、结果摘要面板。
- 本故事不实现 Epic 4 的 BM25 索引、搜索页、索引重建，也不实现 Epic 5 localhost HTTP API。
- 不新增真实外部模型测试依赖；所有失败和重试测试使用 mock/test provider 或本地构造错误。

### Error Code Guidance

- 前置门禁：`model_configuration_incomplete`、`privacy_notice_required`、`api_key_missing`、`model_endpoint_missing`、`model_endpoint_invalid`、`model_endpoint_insecure`
- Provider 调用：`model_client_build_failed`、`model_request_failed`、`model_http_status_failed`、`model_response_too_large`、`model_response_read_failed`、`model_response_not_utf8`、`model_response_json_invalid`
- Schema 校验：`analysis_json_invalid`、`analysis_field_missing`、`analysis_field_invalid`、`analysis_schema_version_unsupported`、`analysis_page_id_mismatch`、`analysis_image_hash_mismatch`、`analysis_image_path_mismatch`、`analysis_document_id_mismatch`、`analysis_page_number_mismatch`、`analysis_text_too_long`、`analysis_array_too_large`、`analysis_retrieval_text_missing`
- Page/job orchestration：`page_image_read_failed`、`page_analysis_already_running`、`page_not_eligible_for_analysis`、`stale_page_analysis_lease_lost`、`page_analysis_interrupted`、`analysis_batch_failed`、`analysis_batch_succeeded_with_failures`

不要随意重命名既有错误 code；测试和后续 UI 文案都应依赖稳定 code。如果某个 code 过粗，应在最接近的边界新增 code，并用测试锁定行为。

### Previous Story Intelligence

- Story 3.4 review fixed a critical batch outcome issue: partial failures must not be persisted as a pure success job. Story 3.5 must preserve that behavior while improving page-level diagnostics.
- Story 3.4 recovery now marks stranded `analysis_pending` pages failed and writes failed `analysis_results`; Story 3.5 should keep recovery retryable and safe.
- Story 3.3 review fixed failure convergence, transactionality, timeout/response-size limits, HTTPS guardrails, duplicate page analysis lease protection, and safe provider summaries. Story 3.5 must not regress any of these protections.
- Story 3.2 validator intentionally rejects unknown fields, limits payload/text/array sizes, and reports paths without raw output. Story 3.5 must not bypass validator or store raw invalid model output.
- Story 3.1 keyring and privacy notice are the model-call boundary; retry UI and services must not read or expose full API keys.
