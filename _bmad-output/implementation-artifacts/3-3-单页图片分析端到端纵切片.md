# Story 3.3: 单页图片分析端到端纵切片

**Status:** done

## Story

As a 本地文档处理用户,  
I want 对单个已渲染页面图片执行多模态分析并得到可信 JSON,  
So that 我可以验证从页面图片、模型 provider、schema 校验到结果入库的核心分析链路。

## Acceptance Criteria

1. **Given** 工作区中存在已导入且页面图片可访问的 `page_record`  
   **When** 用户或 service 请求分析单个页面  
   **Then** 应用应创建分析任务或任务子项，并通过 Job Orchestrator 报告状态  
   **And** 长时间模型调用不得直接在 Tauri command 中执行

2. **Given** 模型配置已完成且用户已确认隐私提示  
   **When** `AnalysisService` 分析页面图片  
   **Then** 系统应通过 `ModelProvider` trait 或等价 provider 边界调用具体模型实现  
   **And** MVP 至少应提供 custom/cloud HTTP provider 及一个测试替身或 mock provider

3. **Given** provider 构建模型请求  
   **When** 请求包含页面图片、prompt、model name 和 endpoint 配置  
   **Then** prompt 应要求模型返回符合 `page_analysis_v1` 的 JSON  
   **And** 请求日志不得记录完整 API key、Authorization header 或未脱敏图片/响应内容

4. **Given** 模型返回符合要求的 JSON  
   **When** schema 校验通过  
   **Then** 应用应将分析结果写入 `analysis_results` 或等价账本记录，包含 `page_id`、schema version、provider、model name、状态、结果 JSON 和时间戳  
   **And** 对应页面分析状态应更新为 `analyzed` 或等价成功状态

5. **Given** 模型返回结果中的 source 或图片字段与当前页面不一致  
   **When** `AnalysisService` 校验结果上下文  
   **Then** 系统应拒绝该结果并记录结构化错误  
   **And** 不得让错误页面身份进入 `analysis_results`

6. **Given** 单页分析成功完成  
   **When** 用户查看工作台或页面详情占位  
   **Then** 用户应能看到该页已分析成功和基础摘要状态  
   **And** 本故事不应实现全文搜索、索引构建或 HTTP 查询行为

### 数据模型建议

`analysis_results` 的最小表结构建议：

```sql
CREATE TABLE IF NOT EXISTS analysis_results (
  analysis_id TEXT PRIMARY KEY,
  page_id TEXT NOT NULL,
  schema_version TEXT NOT NULL,
  provider TEXT NOT NULL,
  model_name TEXT NOT NULL,
  status TEXT NOT NULL CHECK (status IN ('succeeded', 'failed')),
  result_json TEXT,
  error_id TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  FOREIGN KEY (page_id) REFERENCES page_records(page_id),
  FOREIGN KEY (error_id) REFERENCES errors(error_id)
);
```

当前纵切片可以增加 `UNIQUE(page_id)` 或用 repository 覆盖该页 current result；若要支持版本历史，留到 Story 3.6，不要在本故事扩大范围。

### Provider 合同约束

- `ModelProvider` trait 应让 service 可注入 mock provider；不要让 service 直接构造 `reqwest`。
- `CustomHttpModelProvider` 可使用 blocking client，因为 Tauri command 会通过 `spawn_blocking` 调用 service；如果改 async，需要保持 command/service 边界清晰。
- 错误 details 只能包含 endpoint 类别、HTTP status、响应长度、correlation id 等安全摘要；不得包含 API key、Authorization header、完整请求体、完整图片 base64 或完整模型响应。
- 如果 provider 返回 `{ "content": "..." }`，只把 `content` 作为待校验 raw JSON 传给 validator；不要把 provider 的整个响应作为分析结果落库。

### 架构合规（必须遵守）

- 业务逻辑在 `services/`，SQLite 只在 `repositories/`，模型调用能力在 `providers/model/`。[Source: `_bmad-output/planning-artifacts/architecture.md` — Architectural Boundaries]
- Tauri commands 只负责参数映射、`spawn_blocking` 和调用 service；不得直接读写 SQLite、读取图片、构建 HTTP 请求或处理 schema 校验。[Source: `_bmad-output/planning-artifacts/architecture.md` — Module Interaction Patterns]
- SQLite 是主状态源；JSONL 和前端状态不得作为分析状态 source of truth。[Source: `_bmad-output/planning-artifacts/architecture.md` — Core Decisions]
- `page_id` 和 `image_hash` 必须保持分离；搜索/分析结果必须可追溯到文档、页码、页面图片和结构化 JSON。[Source: `_bmad-output/planning-artifacts/architecture.md` — Page Identity Decision Pressure]
- 本故事不得实现批量分析、单文档重新分析、分析 JSONL 导出、BM25 索引、搜索 UI 或 localhost HTTP API。[Source: `_bmad-output/planning-artifacts/epics.md` — Story 3.3/3.4/3.6/4/5 boundaries]
