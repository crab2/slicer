# 产品需求文档 - slicer

**作者:** xq  
**日期:** 2026-05-14  
**文档状态:** 完成  
**产品类型:** Windows 优先的 Rust/Tauri 桌面应用  

## 1. 执行摘要

slicer 是一款本地优先的桌面级文档页面切片与多模态检索应用。用户可以设置本地工作目录，将 PDF、PPT、PPTX、DOC、DOCX 等文件拖入应用，系统将文档逐页转换为图片，并使用图片内容哈希进行唯一命名。随后，应用调用用户配置的多模态大模型接口分析每张图片，生成统一 JSON 元数据。最后，系统基于内置 BM25 索引提供桌面内搜索和 localhost HTTP 查询接口，返回命中的页面 JSON、图片地址、来源文档、页码和相关分数。

第一版以 Windows 为主要交付平台，采用 Rust + Tauri 架构。Office 文档转换通过本机 LibreOffice headless 能力完成，PDF 页面渲染由本地转换模块处理。应用默认将文件、图片、元数据、任务状态和索引全部存放在用户指定的本地目录，不做默认云同步。多模态分析仅在用户配置云端 API 或自定义 HTTP endpoint 后执行，并在界面中明确提示图片会发送至配置的模型服务。

## 2. 产品目标

### 2.1 核心目标

1. 将常见办公文档拆解为页面级图片资产，保留来源文档与页码关系。
2. 使用多模态大模型把页面图片转换为可搜索、可理解、可追溯的结构化 JSON。
3. 在本地构建页面级 BM25 检索能力，并返回页面 JSON 与图片路径。
4. 提供简洁美观的桌面 GUI，让非开发者也能完成导入、转换、分析、搜索和重试。
5. 保留 qmd、wiki search、向量或混合检索的后续扩展接口，但不把它们作为第一版核心依赖。

### 2.2 成功标准

1. 用户可以在 Windows 上完成从设置工作目录到搜索页面结果的完整闭环。
2. 一个 30 页 PPTX 文件导入后，可以稳定生成 30 张页面图片，并为每页生成 JSON 元数据。
3. 搜索关键词能够命中页面标题、摘要、可见文字、主题、关键词和来源文件名。
4. 查询结果必须包含页面 JSON 和图片地址，不允许只返回文本片段。
5. 转换、分析、索引重建任务失败后，用户可以在 GUI 中看到原因并执行重试。
6. 应用重启后，任务状态、文档状态、页面记录和已生成图片仍可恢复。

## 3. 用户与使用场景

### 3.1 目标用户

1. 本地知识库维护者：需要把大量 PDF、PPT、Word 资料拆成页面级知识资产。
2. 研究人员与学生：需要按页面检索课件、论文截图、报告和资料库内容。
3. 企业资料整理者：需要对培训材料、方案文档、销售 PPT、制度文件进行快速检索。
4. 开发者或自动化用户：需要通过本地 HTTP API 获取页面 JSON 和图片路径，接入其他工具链。

### 3.2 关键场景

1. 用户第一次打开应用，选择一个本地工作目录作为资料库根目录。
2. 用户拖入多个 PDF/PPTX/DOCX 文件，点击转换，等待页面图片生成。
3. 用户配置多模态模型 API，点击分析，系统逐页生成结构化 JSON。
4. 用户搜索“多模态检索架构”，应用返回相关页面、图片预览、来源文档和页码。
5. 用户发现某个任务失败，查看失败原因后点击重试。
6. 用户升级分析 schema 或重新导入数据后，点击重建索引。
7. 用户通过 `GET /search` 在其他本地工具中查询页面结果。

## 4. MVP 范围

### 4.1 包含范围

1. Windows 优先桌面应用。
2. Rust + Tauri GUI。
3. 设置和持久化本地工作目录。
4. 支持拖拽或文件选择导入 PDF、PPT、PPTX、DOC、DOCX。
5. PDF 逐页渲染为 PNG 图片。
6. PPT/PPTX/DOC/DOCX 通过本机 LibreOffice headless 转换为 PDF 后再渲染为 PNG。
7. 使用图片内容哈希命名页面图片。
8. SQLite 保存任务、文档、页面、分析状态和索引状态。
9. 文件目录保存原始文档、页面图片、JSONL 元数据和 BM25 索引文件。
10. 云端 API 与自定义 HTTP endpoint 多模态模型配置。
11. 多模态分析输出 `page_analysis_v1` JSON。
12. 模型输出 JSON schema 校验。
13. 分析失败、超时、非法 JSON、API 错误的失败记录与重试。
14. 内置 BM25 检索。
15. 搜索结果返回页面 JSON、图片地址、来源文档、页码和相关分数。
16. 本地 GUI 搜索页。
17. localhost HTTP 查询 API。
18. 索引重建入口。
19. 空状态、加载状态、失败状态、重试状态、索引重建状态。

### 4.2 不包含范围

1. 多用户协作。
2. 云端同步。
3. 权限系统。
4. 复杂页面标注编辑器。
5. 内置本地大模型管理平台。
6. macOS/Linux 正式发布承诺。
7. 向量检索或混合检索作为第一版必选能力。
8. 内置 LibreOffice 打包。
9. 自动处理 PPT 动画语义。
10. 文档内容的人工修订工作流。

## 5. 产品体验要求

### 5.1 信息架构

应用第一屏为工作台，不提供营销页或介绍页。主导航包含：

1. 工作台：导入、转换、分析、任务进度。
2. 搜索：关键词查询、结果列表、图片预览、JSON 查看。
3. 设置：工作目录、LibreOffice 路径、模型配置、图片参数、并发参数。

### 5.2 工作台

工作台必须包含：

1. 当前工作目录显示。
2. 选择或更改工作目录按钮。
3. 文件拖拽区域。
4. 文件选择入口。
5. 任务列表。
6. 每个任务的文件名、类型、页数、转换状态、分析状态、失败原因。
7. 转换按钮。
8. 分析按钮。
9. 单任务重试按钮。
10. 全部失败项重试按钮。
11. 索引重建入口。

### 5.3 搜索页

搜索页必须包含：

1. 搜索输入框。
2. 搜索结果列表。
3. 结果项显示标题、摘要、来源文档、页码、相关分数。
4. 图片预览区。
5. 页面 JSON 查看区。
6. 无结果空状态。
7. 索引不可用或正在重建时的状态提示。

### 5.4 设置页

设置页必须包含：

1. 工作目录路径。
2. LibreOffice 可执行文件路径。
3. 自动检测 LibreOffice 按钮。
4. 模型 provider 名称。
5. API key。
6. Base URL。
7. 自定义 endpoint。
8. Model name。
9. 默认图片 DPI，初始值为 144。
10. 转换并发数。
11. 分析并发数。
12. 隐私提示：启用云端模型时，页面图片会发送到用户配置的模型服务。

### 5.5 视觉风格

1. 简洁、克制、桌面工具感强。
2. 信息密度适中，适合批量任务处理。
3. 不使用营销式大 hero。
4. 不使用过度装饰性渐变背景。
5. 常见操作使用清晰按钮和图标。
6. 错误状态必须明确、可定位、可恢复。

## 6. 功能需求

### FR-001 工作目录设置

用户必须能够选择一个本地目录作为工作目录。应用需要在该目录下创建和维护标准子目录、数据库和索引文件。更改工作目录后，应用应重新加载该目录中的数据库、任务状态和索引状态。

验收标准：

1. 首次启动时，未设置工作目录应显示空状态和选择目录入口。
2. 选择目录后，应用自动初始化目录结构。
3. 重启应用后能记住上一次使用的工作目录。
4. 路径包含空格或中文时可正常使用。

### FR-002 文件导入

用户必须能够通过拖拽或文件选择导入 PDF、PPT、PPTX、DOC、DOCX 文件。系统需要计算原文件哈希并识别重复文档。

验收标准：

1. 支持一次导入多个文件。
2. 不支持的文件类型应被拒绝，并显示原因。
3. 重复文档应提示跳过、重新转换或重新分析。
4. 原始文件应复制或登记到 `originals/`，并与 `document_id` 关联。

### FR-003 文档转换

PDF 文件应直接逐页渲染为 PNG。PPT/PPTX/DOC/DOCX 应先通过本机 LibreOffice headless 转换为 PDF，再逐页渲染为 PNG。

验收标准：

1. 1 页、30 页、300 页 PDF 均能生成正确页数图片。
2. 未检测到 LibreOffice 时，Office 文档转换应失败为可恢复状态，并提示配置路径。
3. LibreOffice 转换超时或失败时，应记录 stderr 摘要或诊断信息。
4. 页面图片生成后应写入页面记录。
5. 默认图片格式为 PNG，默认 DPI 为 144。

### FR-004 图片哈希命名

每张页面图片必须使用图片内容哈希命名，避免因来源文件名、页码或重复导入造成冲突。

验收标准：

1. 相同图片内容生成相同 `image_hash`。
2. 图片文件名格式为 `<image_hash>.png`。
3. `page_id` 第一版与 `image_hash` 保持一致。
4. 同一文档下不同页面不得因文件名冲突覆盖。

### FR-005 多模态模型配置

用户必须能够配置云端 API 或自定义 HTTP endpoint，用于页面图片分析。

验收标准：

1. 支持配置 provider、API key、base URL、custom endpoint、model name。
2. 未配置模型时，分析按钮不可执行或执行前提示配置。
3. 启用云端模型前显示隐私提示。
4. API key 不应出现在日志、错误提示或导出的 JSON 中。

### FR-006 页面分析

系统必须对新生成或标记为需重跑的页面图片调用多模态模型，并生成 `page_analysis_v1` JSON。

验收标准：

1. 只分析新页面或用户明确要求重跑的页面。
2. 正常 JSON 输出应通过 schema 校验并入库。
3. 非法 JSON、超时、API 错误应进入失败状态。
4. 失败页面可单独重试。
5. 支持单文档重新分析。

### FR-007 元数据保存

系统必须同时保存结构化数据库记录和可读 JSONL 元数据。

验收标准：

1. SQLite 保存任务、文档、页面、分析状态。
2. `metadata/pages.jsonl` 保存页面级 JSON 记录。
3. 数据库与 JSONL 中的 `page_id`、`document_id`、`image_path` 保持一致。
4. 应用异常退出后，重启能恢复已完成和失败状态。

### FR-008 BM25 检索

系统必须基于页面分析结果构建 BM25 索引。

验收标准：

1. 索引文本包含标题、摘要、可见文字、主题、关键词、来源文件名。
2. 搜索关键词可以命中中文内容。
3. 搜索结果按相关性排序。
4. 结果返回相关分数。
5. BM25 索引不可用时，GUI 应显示明确状态。

### FR-009 查询返回

搜索结果必须返回页面 JSON 和图片地址。

验收标准：

1. GUI 搜索结果可打开页面图片预览。
2. GUI 可查看对应页面 JSON。
3. HTTP API 返回 `page_id`、`score`、`image_path`、页面 JSON 摘要和来源信息。
4. 不允许只返回文本片段而缺少图片地址。

### FR-010 索引重建

用户必须能够按全部页面重建 BM25 索引。

验收标准：

1. 重建索引不删除原图片。
2. 重建索引不删除页面 JSON。
3. 重建失败不破坏上一个可用索引。
4. GUI 显示重建进度、成功状态和失败原因。

### FR-011 本地 HTTP API

应用必须提供 localhost HTTP API，供外部本地工具查询。

验收标准：

1. API 默认仅监听 localhost。
2. `GET /health` 返回服务状态。
3. `GET /search?q={query}&limit={n}` 返回搜索结果。
4. `GET /pages/{page_id}` 返回单页 JSON 与图片地址。
5. `GET /documents/{document_id}` 返回文档元数据与页面列表。
6. `POST /indexes/rebuild` 触发索引重建任务。

### FR-012 扩展检索接口

系统应保留 `SearchProvider` 或等价 adapter 概念，为 qmd、wiki search、向量或混合检索扩展预留接口。

验收标准：

1. 第一版默认 provider 为内置 BM25。
2. 查询接口不应硬编码为只能支持 BM25。
3. qmd/wiki search 不作为 MVP 必须完成项。

## 7. 数据结构与接口

### 7.1 工作目录结构

```text
workspace/
  originals/
  pages/
    <document_id>/
      <image_hash>.png
  metadata/
    pages.jsonl
  indexes/
    bm25/
  app.db
```

schema 名称：`page_analysis_v1`

```json
{
  "page_id": "7f9a2c91b44d18e2",
  "image_hash": "7f9a2c91b44d18e2",
  "image_path": "D:/KnowledgeBase/pages/a83c91d0e4b72a11/7f9a2c91b44d18e2.png",
  "source": {
    "document_id": "a83c91d0e4b72a11",
    "original_filename": "AI产品方案.pptx",
    "original_path": "D:/KnowledgeBase/originals/a83c91d0e4b72a11.pptx",
    "page_number": 12,
    "total_pages": 30,
    "document_type": "pptx"
  },
  "analysis": {
    "title": "多模态检索系统架构",
    "summary": "该页展示了文档转图片、视觉理解、索引构建和查询返回的整体流程。",

    "provider": "custom_http",

  },
  "schema_version": "page_analysis_v1"
}
```

### 7.3 HTTP API

#### GET /health

返回应用和索引状态。

```json
{
  "status": "ok",
  "workspace_ready": true,
  "index_status": "ready"
}
```

#### GET /search

请求：

```text
GET /search?q=多模态检索&limit=20
```

响应：

```json
{
  "query": "多模态检索",
  "limit": 20,
  "results": [
    {
      "score": 0.912,
      "page_id": "7f9a2c91b44d18e2",
      "image_path": "D:/KnowledgeBase/pages/a83c91d0e4b72a11/7f9a2c91b44d18e2.png",
      "page": {
        "title": "多模态检索系统架构",
        "summary": "该页展示了文档转图片、视觉理解、索引构建和查询返回的整体流程。",
        "topics": ["多模态", "文档解析", "检索"]
      },
      "source": {
        "document_id": "a83c91d0e4b72a11",
        "original_filename": "AI产品方案.pptx",
        "page_number": 12,
        "total_pages": 30
      }
    }
  ]
}
```

#### GET /pages/{page_id}

返回完整页面 JSON 和图片地址。

#### GET /documents/{document_id}

返回文档元数据、页面数量、转换状态、分析状态和页面列表。

#### POST /indexes/rebuild

触发后台索引重建任务，返回任务 ID 和初始状态。

## 8. 存储需求

### 8.1 SQLite

SQLite 至少应保存：

1. 应用设置。
2. 文档记录。
3. 页面记录。
4. 转换任务。
5. 分析任务。
6. 分析错误。
7. 索引状态。

### 8.2 文件存储

文件系统至少应保存：

1. 原始文件。
2. 中间 PDF 文件，是否长期保留由实现配置决定。
3. 页面 PNG 图片。
4. `metadata/pages.jsonl`。
5. BM25 索引文件。

### 8.3 一致性要求

1. 数据库中的图片路径必须指向真实存在的图片文件。
2. JSONL 中每条记录必须能通过 `page_id` 对应到 SQLite 页面记录。
3. 索引可以重建，不能成为唯一元数据来源。

## 9. 非功能需求

### 9.1 性能

1. 30 页 PPTX 在正常 LibreOffice 环境下应能完成转换和页面记录生成。
2. 300 页 PDF 转换时 GUI 不得卡死。
3. 长任务必须后台执行，并持续更新进度。
4. 搜索接口在普通本地资料库规模下应在可感知的短时间内返回结果。

### 9.2 稳定性

1. 转换、分析、索引重建任务都必须可失败、可记录、可恢复。
2. 应用异常关闭后，重启不能丢失已完成页面记录。
3. 重建索引失败不能破坏已有可用索引。

### 9.3 安全与隐私

1. 所有数据默认保存在用户选择的本地工作目录。
2. 不做默认云同步。
3. API key 不写入普通日志。
4. 调用云端模型前必须让用户知道图片会发送到配置的模型服务。
5. localhost API 默认不监听公网地址。

### 9.4 兼容性

1. 第一版仅承诺 Windows 优先。
2. 路径包含中文、空格时必须可用。
3. 支持中文文件名和中文页面内容检索。
4. macOS/Linux 作为后续兼容方向，不进入第一版验收承诺。

### 9.5 可扩展性

1. 检索层需要保留 provider/adapter 抽象。
2. 模型调用层需要保留 provider/endpoint 抽象。
3. 页面 JSON schema 需要带版本号，支持后续升级。

## 10. 任务状态模型

### 10.1 文档状态

文档状态应至少包含：

1. `imported`
2. `converting`
3. `converted`
4. `conversion_failed`
5. `analyzing`
6. `analyzed`
7. `analysis_failed`
8. `indexed`
9. `index_failed`

### 10.2 页面状态

页面状态应至少包含：

1. `image_created`
2. `analysis_pending`
3. `analysis_running`
4. `analysis_succeeded`
5. `analysis_failed`
6. `indexed`

### 10.3 错误信息

错误记录应至少包含：

1. 错误类型。

## 13. 验收标准

MVP 可以验收发布的最低条件：

1. Windows 环境下应用可以启动、选择工作目录并持久化设置。
2. PDF、PPTX、DOCX 至少各有一个样例文件完成端到端流程。
3. 30 页 PPTX 能生成 30 张图片。
4. 图片文件名使用内容哈希，不使用原文件名或页码直接命名。
5. 每页图片都有对应页面 JSON。
6. 搜索能返回页面 JSON、图片地址、来源文档和页码。
7. 本地 HTTP API 可通过 localhost 查询。
8. 分析失败和转换失败均可在 GUI 中看到并重试。
9. 索引可重建，失败时不破坏已有元数据。
10. 应用重启后，已导入文档、已生成页面和任务状态仍存在。

## 14. 假设与默认值

1. 第一版仅承诺 Windows 优先。
2. Office 转换不内置 LibreOffice，仅检测并调用本机 LibreOffice。
3. 默认图片格式为 PNG。
4. 默认渲染 DPI 为 144。
5. 多模态分析第一版通过云端 API 或自定义 HTTP endpoint，不要求本地视觉模型。
6. 检索第一版以内置 BM25 为主。
7. qmd、wiki search、向量或混合检索作为后续扩展接口，不作为 MVP 必须完成项。
8. 存储采用 SQLite + 文件目录。
9. `page_id` 第一版默认等于 `image_hash`。
10. API 默认仅监听 localhost。

## 15. 开放问题

以下问题不阻塞 MVP PRD，但应在架构设计或实现前进一步确认：

1. 中间 PDF 是否长期保留，还是转换成功后清理。
2. 图片哈希使用完整 SHA-256 还是截断显示，数据库中是否保留完整哈希。
3. BM25 分词方案如何针对中文优化。
4. 模型分析 prompt 是否需要针对 PPT、论文、表格、架构图设置不同模板。
5. 本地 HTTP API 是否需要用户可配置端口。
6. API 是否需要最小访问令牌，防止其他本地进程误用。
